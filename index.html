嵌入式的三种模式

这一页从三种典型嵌入式模式切入：Bare-metal 强调直接控制硬件，FreeRTOS 强调用调度器把任务关系组织清楚，Linux 强调建立完整内核与用户空间。后面的章节不是孤立知识点，而是在同一条学习链里反复比较这三种模式在启动、中断、时钟、存储、驱动和调试上的差异。

STM32 Bare-metal

重点看短链路、寄存器直达、状态机组织以及中断与主循环怎样配合。

FreeRTOS

重点看中断如何交接给任务、同步对象怎样组织事件流，以及调度器如何安排运行顺序。

Linux

重点看完整内核栈、驱动模型、用户态路径与系统级抽象是怎样一步步建立起来的。

阅读方式

建议先把 01-08 当成一条连续知识链来读：先理解系统怎样启动，再理解事件如何进入中断、怎样穿过缓冲与调度，最后再回到驱动分层、地址模型和调试定位。

每一章都先看“为什么学习这个主题”，把问题意识立住；再看三栏对比，观察同一个问题在 Bare-metal、FreeRTOS、Linux 中分别落在哪一层；最后用底部的一行对比与小例子把抽象概念落回真实工程动作。

Bare-metal：短链路，尽快进入主逻辑

适合先抓“启动快、路径短、直接控硬件”这条主线，建立对寄存器、中断和主循环协作的直觉。

FreeRTOS：交给调度器，再由任务展开

适合重点观察 ISR 如何交接给队列、信号量与任务，以及系统怎样从单线程启动切到正式调度阶段。

Linux：建立完整内核与用户空间

适合把驱动模型、文件系统、虚拟内存和用户态接口放在一条长链路里理解，不再把它误看成“只是 MCU 多几层”。

01 上电启动流程对比｜STM32 Bare-metal vs FreeRTOS vs Linux

学习流程图：先看为什么学，再看三条启动链，最后看差异和例子

为什么学习这个主题

这张图是学习三种系统差异的入口。
先看清“谁先启动、谁后启动、必须先做什么、做完后系统进入什么状态”。
它要回答三个常见困惑：
1) 为什么裸机很快跑到 main()；
2) 为什么 FreeRTOS 不是一直停在主循环；
3) 为什么 Linux 的驱动初始化位置最靠后却最完整。

裸机 Bare-metal

短链路：尽快进入 main() 和主循环

bare
1. 上电复位

MCU 复位，PC 准备从复位向量进入启动入口。

2. 取复位向量并装载主栈

CPU 从向量表拿到初始栈顶和 Reset_Handler 入口，MSP 同时完成装载。

3. 进入启动文件

汇编启动代码开始搭建最基础运行环境。

4. 拷贝 .data 清零 .bss

把已初始化数据搬到 SRAM，未初始化数据清零。

5. 配置时钟与系统基础环境

调用 SystemInit 等基础初始化，设置时钟树、Flash 等待周期和核心运行环境。

6. 初始化运行库

按工程需要完成 libc、全局变量或构造相关准备，然后跳转到 main()。

7. 进入 main()

开始执行应用入口，后续初始化和业务逻辑从这里正式展开。

8. 初始化外设

按需求初始化 GPIO、UART、SPI、ADC、TIM 等具体外设资源。

9. 进入 super loop

主循环持续运行，轮询状态并驱动后续业务逻辑。

FreeRTOS

中等链路：启动后把控制权交给调度器

rtos
1. 上电复位

MCU 复位，CPU 准备从向量表给出的启动入口开始执行。

2. 取复位向量并装载主栈

CPU 取到初始栈顶和 Reset_Handler 入口，MSP 同时完成装载。

3. 进入启动文件

汇编启动代码开始执行，准备建立最基础运行环境。

4. 完成基础运行环境

拷贝 .data、清零 .bss，执行 SystemInit 和必要运行库初始化。

5. 进入 main()

开始进入应用入口，FreeRTOS 相关初始化从这里正式展开。

6. 初始化外设与 RTOS 运行前配置

准备串口、SPI、DMA 等底层资源，并设置与 RTOS 配合的中断优先级规则。

7. 创建内核对象

队列、信号量、事件组等同步通信对象先建立起来。

8. 创建任务

采集、通信、显示、存储等任务被创建出来，等待调度器接管。

9. 启动调度器并进入任务阶段

vTaskStartScheduler() 之后系统控制权交给内核，中断发事件、任务承担主要处理。

Linux

长链路：建立完整内核和用户空间环境

linux
1. 上电复位

SoC 上电复位，CPU 从固化启动入口开始执行最早期代码。

2. BootROM

芯片内部 BootROM 判断启动介质，并准备装载下一阶段 BootLoader。

3. SPL / 第一阶段 BootLoader

完成早期时钟、DDR 等基础初始化，让系统具备继续装载更大镜像的能力。

4. U-Boot

加载内核、设备树和启动参数，并决定根文件系统等启动信息。

5. 跳转到内核入口

BootLoader 把控制权交给内核入口，开始进入 Linux 启动阶段。

6. 内核解压并进入 start_kernel()

内核完成自解压，正式进入主启动函数，开始建立核心运行环境。

7. 建立页表与开启 MMU

内核建立基础地址映射，使虚拟地址和内存保护机制开始可用。

8. 初始化内存管理与核心子系统

页分配器、伙伴系统、调度、中断等核心基础设施逐步建立起来。

9. 解析设备树

识别板级硬件资源、中断号、寄存器区和总线设备信息。

10. 初始化驱动框架

driver core、bus 和各类子系统逐步就绪，准备开始枚举设备与匹配驱动。

11. 驱动 probe 与挂载根文件系统

设备完成匹配与 probe，系统继续建立文件系统与用户空间运行条件。

12. 启动 init

第一个用户空间进程启动，系统服务与应用开始逐步拉起。

对比观察

裸机启动链最短，目标是尽快把 MCU 拉到“可直接控制寄存器和外设”的状态；
FreeRTOS 在裸机基础上多了一层“把系统从单线程启动阶段切换到任务调度阶段”的过程；
Linux 启动链最长，因为它不仅要让 CPU 跑起来，还要把 MMU、页表、设备树、驱动模型、文件系统和用户空间一起建立起来。

小例子说明

例子：让串口打印 hello。
裸机：main() 里初始化 UART 后直接写发送寄存器或调用发送函数；
FreeRTOS：系统启动后由串口任务拿到运行机会，再通过队列/任务调度组织发送；
Linux：应用调用 write()，数据先进入系统调用路径，再由 tty/uart 驱动链路发给 UART 控制器。

02 中断机制对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张聚焦“中断来了以后到底谁先处理、谁后接手、谁负责把数据继续往上送”。先建立三条中断处理链路，再看谁做重活、数据如何流动、上下文怎样切换。

为什么学习这个主题

因为三种系统最大的分水岭之一，就在于中断来了之后到底谁先处理、谁后处理、谁负责把数据交给后续
代码。学清这一张图，后面再看驱动、调度、缓存、实时性时就不会混淆。

裸机 Bare-metal

路径短，很多工作直接在 ISR 里完成。

bare
1. 中断源触发

外设事件发生后，NVIC 准备响应这次中断请求。

2. CPU 进入 ISR

处理器保存现场并跳转到对应中断服务例程。

3. ISR 直接读状态

中断里通常立刻读取状态寄存器、拿走关键数据并清中断标志。

4. 数据先写回本地缓冲

结果常直接写入全局变量、环形缓冲或简单状态结构。

5. ISR 顺手做部分业务判断

为了快，很多项目会在 ISR 里顺手做一部分拼包、计数或结束判断。

6. 返回主循环继续消费

退出中断后再由主循环接着处理剩余业务。

FreeRTOS

ISR 先交接，再由高优先级任务继续做重活。

rtos
1. 中断源触发

外设事件到来后，CPU 先进入对应 ISR，上下文切换仍由硬件和内核规则共同约束。

2. ISR 读取最关键状态

中断里只先拿最急的数据或状态，避免在 ISR 里停留太久。

3. 通过内核对象完成交接

Queue、Semaphore、Event 或任务通知承担中断到任务的交接角色。

4. 判断是否唤醒高优先级任务

如果事件很关键，ISR 会请求让更高优先级任务尽快接手。

5. 调度器安排任务接管

PendSV 等机制会在合适时机切换到被唤醒的任务。

6. 任务在线程上下文做重活

真正的拼包、计算、协议处理和业务逻辑放到任务里完成。

Linux

中断被内核拆成上半部和下半部，分工更细。

linux
1. 设备发出 IRQ

硬件先通过中断线把事件送到 SoC 和内核的中断入口。

2. 进入 generic IRQ 框架

体系结构相关入口会把控制权交给 Linux 的通用中断处理框架。

3. hardirq 先做最急动作

最紧急的确认、取状态和清中断动作先在上半部完成。

4. 后续处理后移到下半部

softirq、threaded IRQ 或 workqueue 承担更重、更长的后续工作。

5. 驱动更新缓冲并唤醒等待者

驱动把数据写入内核缓冲区，并继续唤醒上层等待路径。

6. 更高层子系统或用户态继续消费

TTY、网络栈、输入子系统或用户空间接口最终接走这次中断带来的数据。

对比观察

看谁做重活：裸机是 ISR 自己做得多；FreeRTOS 是 ISR 先交接、任务再接手；Linux 是 hardirq 只做最急的部分，真正复杂的处理尽量后移到更安全、更可调度的上下文。看数据怎么流：裸机常直接写全局变量或环形缓冲；FreeRTOS 常写队列、信号量或任务通知；Linux 常写内核缓冲区并唤醒等待队列，再让更高层子系统或用户态读取。

小例子说明

以串口接收一帧数据为例。裸机里，串口中断一来，ISR 可能把字节逐个搬进环形缓冲，同时顺手判断一帧是否接收完成。FreeRTOS 里，ISR 常只
把新字节或完成事件送入队列，再唤醒串口处理任务，让任务去拼包和校验。Linux 里，串口驱动的硬中断先快速确认状态，后续再由内核缓冲和 TTY 路
径把数据送到 read、poll 或 select 这类接口，最终用户态程序再拿到完整数据。

03 存储系统对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张聚焦“数据存到哪里、怎样真正落到介质”。先看为什么要学存储路径，再看三条典型数据落盘链路，最后用一行对比和一个例子把差异串起来。

为什么学习这个主题

因为很多人一提存储就只想到 Flash、栈、堆这些静态名词，却不知道一次真实的数据保存会经过哪些层。学这一张图，是为了看清裸机怎样直接写介质，FreeRTOS 怎样让任务之间先传递数据，再由存储任务落盘，以及 Linux 怎样把一次写入拆成系统调用、页缓存、文件系统和驱动的完整路径。

裸机 Bare-metal

路径短，上层常直接接触底层资源。

bare
1. 固定内存视角

程序先在固定内存视角下运行，代码在 Flash，全局和静态数据在 SRAM，临时数据通常先落在 RAM 中。

2. 应用组织待保存数据

应用先把参数、日志或采样结果整理成可保存的数据结构，决定这次要写什么内容。

3. 直接调用底层驱动

需要保存时，程序直接调用 SPI、I2C、SDIO 或片上 Flash 驱动，把命令、地址和数据交给介质控制器。

4. 介质执行写入 / 擦除

具体存储芯片按页、扇区或块完成写入与擦除，这一步通常离应用非常近。

5. 一致性与恢复自己承担

掉电一致性、双备份、版本恢复和擦写寿命保护，往往都需要应用层自己设计。

6. 再从介质读回并解析

读取时也是短链路：驱动把数据搬回 RAM，应用自己做校验、解析并决定后续处理。

FreeRTOS

事件先交给内核对象，再由任务继续处理。

rtos
1. 任务先产生数据

采样、通信、显示等任务先分别形成自己的数据记录，数据不再只在单一数组里流动。

2. 进入队列或缓冲对象

数据通常先进入 Queue、Message Buffer、Stream Buffer 等内核对象，等待后续任务接手。

3. 专用存储任务接手

更常见的做法是由专门的存储任务统一处理写介质请求，避免多个任务直接争用同一设备。

4. 通过文件层或磁盘层下沉

如果项目接入 FatFs，存储任务会经由文件接口和磁盘 I/O 适配层，把请求继续下发给底层驱动。

5. DMA / 中断完成后再唤醒

介质忙或 DMA 未完成时，请求方可以阻塞等待，传输结束后再由中断或回调唤醒相关任务。

6. 看清数据最终归属

真正重要的是理清数据在哪个任务里产生、在哪个缓冲里排队、最后由谁真正写到设备上。

Linux

路径更长，但系统层次和抽象能力更强。

linux
1. 用户态发起存储请求

用户程序先通过 fopen、write、fwrite 等接口发出读写请求，看到的是文件或设备节点而不是扇区地址。

2. VFS 路由到具体对象

请求进入内核后先经过 VFS，再根据文件描述符路由到具体文件系统、字符设备或块设备实现。

3. Page Cache 先接住数据

普通文件写入常先进入 Page Cache，数据会先成为脏页，而不是立刻落到真实硬件。

4. 文件系统转换逻辑操作

文件系统层把逻辑文件操作转换成目录项、元数据和数据块更新，并维护一致性策略。

5. 块层 / MTD / 驱动继续下沉

更底层再由块层、MTD 或控制器驱动组织 I/O 请求，配置 DMA、发命令并等待中断完成。

6. 多层之后真正落盘

Linux 的一次保存，本质上是用户态请求经过多层缓存和组织后，最终才真正落到介质上。

对比观察

裸机更像应用直接管理 RAM 与介质，路径最短，但一致性和恢复策略基本都靠自己设计。FreeRTOS 在中间加入任务、队列和专用存储任务，让数据先有序流动，再统一写入设备。Linux 则把一次写入拆成用户态请求、内核缓存、文件系统、块层或 MTD 层以及驱动控制器的完整栈式路径。

小例子说明

例子：保存一条传感器记录。裸机通常在采样完成后直接调用 SPI Flash 写函数；FreeRTOS 常让采样任务把记录投递到队列，再由存储任务写入 SD 卡；Linux 里应用可能只调用 fwrite，数据先进入页缓存，再由文件系统和底层驱动决定真正的写盘时机。

04 时钟模块对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一章单独把时钟模块抽出来讲。因为三种模式虽然都会碰到时钟源、PLL、分频和外设时钟，但它们对“系统时基”和“谁来管理时钟”的理解完全不同。

为什么学习这个主题

很多培训阶段同学会把时钟只当成 RCC 配置题，但在真实项目里，时钟不仅决定主频和外设速率，还会影响 delay、调度节拍、功耗控制以及驱动能不能正常工作。单独把这一章补上，能把“时钟树”和“系统运行节奏”这两件事真正接起来。

裸机 Bare-metal

自己把 RCC 和时钟树手动拉起来

bare
1. 选定时钟源

先确认内部 RC、外部晶振 HSE 或其他时钟源谁作为系统基础输入。

2. 配置 PLL 与主频

根据芯片手册设置 PLL 倍频、分频，让核心频率进入目标工作范围。

3. 切换系统时钟

把 SYSCLK 切到目标源，同时确认切换完成标志和稳定状态。

4. 设置 AHB / APB 分频

继续给总线分频，保证 CPU、内存和外设都工作在安全频率范围内。

5. 给外设单独开时钟

GPIO、USART、SPI、TIM、ADC 等外设还要分别打开对应时钟门控。

6. 自己维护系统时基

delay、超时、定时任务和节拍推进通常要由 SysTick、TIM 或轮询逻辑自己承担。

FreeRTOS

底层时钟树配好后，还要建立 RTOS 系统节拍

rtos
1. 先完成底层时钟树

和裸机一样，先把 HSE、PLL、总线分频和外设时钟配置到位。

2. 建立 RTOS 时基来源

SysTick 或专用定时器会被配置成内核 tick 的来源，成为调度节拍基础。

3. 配合中断优先级规则

与 tick 相关的中断优先级要和 FreeRTOS 的可调用中断规则匹配起来。

4. 延时与超时开始依赖 tick

vTaskDelay、超时等待、软件定时器等机制都建立在系统节拍之上。

5. 外设时钟与任务节奏协同

外设采样频率、DMA 传输节奏和任务唤醒节奏要配合，否则容易抖动或丢数。

6. 低功耗时还要考虑 tickless

如果项目要省电，还要继续处理 tickless idle、唤醒时钟恢复和定时精度问题。

Linux

时钟被抽象成内核资源，由框架统一管理

linux
1. BootLoader 先拉起基础时钟

最早期时钟、DDR 时钟和基础运行频率通常先由 BootLoader 准备好。

2. 设备树描述时钟关系

设备树会描述 clock source、父子关系、频率和设备所依赖的时钟输入。

3. 内核进入 Common Clock Framework

Linux 把时钟抽象进 CCF，由框架统一管理 rate、parent、enable 等关系。

4. 驱动通过 clk API 获取时钟

设备驱动通常不会直接碰寄存器，而是通过 clk_get、clk_prepare_enable 等接口申请时钟。

5. 运行时动态调频与门控

设备空闲时可关钟，忙时再开钟，某些平台还会动态调节频率以平衡性能和功耗。

6. 定时能力交给内核时钟体系

高精度定时器、jiffies、hrtimer、workqueue 和调度器共同承担系统节拍与超时管理。

对比观察

裸机更像“自己把时钟树手工拉起来”；FreeRTOS 是“先把底层时钟树配好，再让 SysTick 和调度器接管系统节拍”；Linux 则进一步把时钟抽象成 Common Clock Framework 里的资源，由设备树、驱动和内核框架共同管理。也就是说，越往后，时钟越不只是 RCC 初始化，而是系统级资源管理。

小例子说明

比如同样是一个串口接收超时功能：裸机常常自己配 TIM 或 SysTick 算时间；FreeRTOS 更可能用 tick、软件定时器或任务延时去组织超时；Linux 则往往交给内核定时器、工作队列或驱动框架中的超时机制。它们都和“时钟”有关，但落点已经完全不同。

05 缓存与缓冲机制对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张聚焦“数据先暂存在哪里、为什么要暂存、谁负责继续往下传”。重点是把 cache 和 buffer 明确拆开，别再混成一个概念。

为什么学习这个主题

因为“缓存”和“缓冲”最容易被混着讲。学这一张图，是为了真正分清缓冲主要解决速度差和时序解耦，缓存主要解决重复访问成本；同时也要看清三种系统里这些机制由谁来维护，复杂度为什么会越来越高。

裸机 Bare-metal

路径短，上层常直接接触底层资源。

bare
1. 外设或 DMA 先把数据接住

串口、ADC 或其他外设先通过寄存器、DMA 或数组，把最先到来的数据暂存在 RAM 中。

2. ISR 只做快速搬运

中断里通常只推进环形缓冲指针、记录状态或标记缓冲已满，先保证数据别丢。

3. 主循环再慢慢消费

主循环之后再从缓冲区取数据、拼帧、计算或显示，缓冲区承担了时间解耦的职责。

4. 环形 / 双缓冲避免堵塞

环形缓冲和双缓冲常用来隔开快外设和慢处理逻辑，减少覆盖和丢样风险。

5. DMA 一致性自己管理

如果 DMA 参与收发，就必须自己想清楚写满时机、读取时机以及读写指针是否会相撞。

6. 关键是读写节奏

学习这一栏时，重点不是背名词，而是看清谁在写、谁在读、何时覆盖、何时清空。

FreeRTOS

事件先交给内核对象，再由任务继续处理。

rtos
1. ISR 先接住数据

外设中断先把最紧急的数据接住，避免在中断里做过重的后续业务。

2. 写入 Queue / Stream Buffer

接到的数据通常先进入 Queue、Stream Buffer、Message Buffer 或任务通知机制。

3. 唤醒对应处理任务

数据一旦入队，就可以唤醒对应任务，把后续解析、拼包、滤波等工作转交出去。

4. 任务在线程上下文消费

真正的业务处理放在线程上下文完成，这样既便于维护，也更容易控制执行时长。

5. 多任务共享时再加同步

若多个任务共用同一缓冲，还要通过互斥锁、信号量或事件组保证共享访问正确。

6. 缓冲区反过来决定节奏

缓冲区大小、阻塞时间和任务优先级会一起决定系统能不能扛住突发数据。

Linux

路径更长，但系统层次和抽象能力更强。

linux
1. 数据先进入对应缓存层

文件数据、网络数据和设备 I/O 往往先进入各自的缓存或缓冲结构，而不是直接被应用看到。

2. 多种缓存 / 缓冲分工并存

Linux 同时存在 CPU Cache、Page Cache、Socket Buffer、I/O Buffer 和 DMA Buffer，它们职责并不相同。

3. 数据在多层之间流转

同一份数据可能先后经过页缓存、协议缓冲和设备缓冲，甚至经历映射与再次复制。

4. 一致性和拷贝次数要控制

DMA Buffer 需要处理 cache 一致性，系统还会通过 mmap 或零拷贝等方式尽量减少重复搬运。

5. 性能分析绕不开这些层

很多性能问题都不是某个函数慢，而是数据在缓存和缓冲层之间走得太重、太绕。

6. 先分清 Cache 与 Buffer

真正关键的是分清哪一层在加速重复访问，哪一层在承接数据过渡，不要把它们混成一个概念。

对比观察

裸机主要靠应用自己维护缓冲，目标是先把数据接住别丢；FreeRTOS 的缓冲同时承担暂存数据和驱动任务协同两种职责；Linux 则既有为了提速的缓存，也有为了跨层传输的缓冲，而且还要处理一致性和拷贝次数问题。

小例子说明

例子：ADC 连续采样一批点。裸机通常让 DMA 先把样本写入数组，主循环稍后处理；FreeRTOS 会让 DMA 完成后通知采样任务，再把数据交给分析任务；Linux 中驱动会先把 DMA 数据放进内核缓冲，用户态再通过 read 或 mmap 拿到样本。

06 并发调度与实时性对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张要解决“多个事情同时发生时系统先顾谁”的问题。重点从中断转到调度与实时性，观察三种系统怎样安排关键工作与后台工作。

为什么学习这个主题

因为很多系统不是只有一个功能同时在跑。学这一张图，是为了看清裸机为什么靠主循环结构决定响应速度，FreeRTOS 为什么靠优先级和阻塞唤醒来保证关键任务先运行，以及 Linux 为什么默认更偏向综合能力而不是最短确定性响应。

裸机 Bare-metal

路径短，上层常直接接触底层资源。

bare
1. 主循环承担常规推进

采样、通信、显示和控制等逻辑通常都在一条 super loop 里轮着执行。

2. 中断抢占处理急事

当更紧急的事件到来时，中断会先打断主循环，于是系统形成主循环加中断的并发模式。

3. 逻辑必须拆成短状态机

为了不拖慢全局，每段工作都要尽量拆短、尽快返回，否则整个系统都会被卡住。

4. 顺序受代码结构影响

除中断优先级外，普通逻辑谁先得到更多 CPU，往往取决于主循环书写顺序和每段代码长度。

5. 实时性主要来自路径短

裸机的实时性很多时候不是因为调度先进，而是因为执行路径短、层次少、控制直接。

6. 风险是一个环节拖慢全局

一旦某一步忙等太久、打印太久或轮询太久，整个系统的响应都会一起变慢。

FreeRTOS

事件先交给内核对象，再由任务继续处理。

rtos
1. 先按职责拆成任务

RTOS 把采样、通信、显示、控制和后台处理拆成独立任务，每个任务有自己的栈和状态。

2. 就绪任务按优先级选择

多个任务同时可运行时，调度器会优先让更高优先级任务获得 CPU。

3. 没事就阻塞等待事件

任务不必忙轮询，它们可以阻塞在队列、信号量、事件组或延时上，等事件发生后再被唤醒。

4. ISR 通知关键任务接手

中断到来时，ISR 可以快速发通知，把真正的业务处理转交给高优先级任务。

5. 调度器完成上下文切换

PendSV 等机制会在合适时机完成切换，让关键任务尽快接手系统控制权。

6. 实时性来自可安排的优先级体系

RTOS 的价值不只是快，而是更容易把关键事情稳定地安排到前面执行。

Linux

路径更长，但系统层次和抽象能力更强。

linux
1. 先进入更大的并发世界

Linux 同时管理用户进程、线程、内核线程、中断上下文和后台工作，系统并发规模更大。

2. 调度类按策略安排运行

普通任务通常由 CFS 等调度类管理，实时任务则可使用更高实时优先级的策略。

3. 事件等待机制更丰富

等待和唤醒不再只靠单一队列，而是可以通过 wait queue、poll、epoll、signal、futex 等机制完成。

4. 设备事件跨多层调度

一个硬件事件往往会经过中断、底半部、工作队列，再一路唤醒到用户态或其他内核路径。

5. 更擅长复杂业务并发

Linux 默认更适合图形、网络、文件系统和多个应用同时运行的复杂软件场景。

6. 强实时要额外增强

如果要进一步追求强实时，通常还会引入 PREEMPT_RT 等增强手段，而不是只靠默认配置。

对比观察

裸机的实时性主要靠路径短和人工拆状态机；FreeRTOS 的实时性来自优先级调度、阻塞唤醒和中断配合；Linux 默认更强调复杂系统的综合能力，实时性可以增强，但理解它必须放在多进程、多层次调度的背景下。

小例子说明

例子：系统同时要做采样、串口输出和按键处理。裸机通常在主循环里轮询，再让中断打断紧急事件；FreeRTOS 会把采样任务设为高优先级，把串口和按键放到合适优先级；Linux 里则可能由驱动、中断线程和用户态线程一起参与，最终由调度器决定谁先获得 CPU。

07 虚拟内存与物理内存对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张专门补上很多 STM32 学习者最陌生的部分：为什么裸机几乎只谈物理地址，而 Linux 一定要谈 MMU、页表和虚拟地址。

为什么学习这个主题

因为很多人在 MCU 世界里几乎不接触虚拟内存，一到 Linux 就容易断层。学这一张图，是为了把“直接看物理地址”和“通过 MMU 做地址抽象”这两个世界接起来，看清 Linux 为什么必须这样设计，而裸机和大多数 FreeRTOS 为什么通常不需要。

裸机 Bare-metal

路径短，上层常直接接触底层资源。

bare
1. 直接面对物理地址

程序通常直接看到真实物理地址，Flash、SRAM 和外设寄存器都在固定位置上。

2. 存储区域位置固定

代码、全局变量、栈和外设空间都靠链接脚本和芯片手册明确安排。

3. 没有完整页表和进程隔离

裸机没有 Linux 那样的页表体系，也没有进程级虚拟地址空间。

4. MPU 只能做有限保护

若芯片带 MPU，更多只是限制部分区域访问权限，而不是建立完整虚拟内存模型。

5. 写坏内存往往影响全局

一旦某段地址被错误覆盖，影响的通常是整个系统，而不是某个独立进程。

6. 模型简单但抽象能力弱

这种模型直观、门槛低，但隔离能力和地址抽象能力都比较有限。

FreeRTOS

事件先交给内核对象，再由任务继续处理。

rtos
1. 仍然以物理地址视角为主

大多数 FreeRTOS 系统仍运行在 MCU 上，本质上依然是共享物理地址空间。

2. 多任务共享同一套内存

多个任务只是轮流使用 CPU，并不代表每个任务都有自己的独立虚拟地址空间。

3. 任务切换不等于地址空间切换

RTOS 擅长的是让任务切换有秩序，但通常并不会伴随 Linux 那样的地址空间切换。

4. MPU 可做有限隔离

某些 FreeRTOS 变体可以借助 MPU 为任务做有限保护，但能力依然远弱于页式虚拟内存。

5. 结构更规整但模型没变

RTOS 会让代码组织更清楚，却通常不会改变大家仍共享同一套物理地址的事实。

6. 别把多任务误解成多地址空间

理解这一栏时，最关键的是把“任务多”与“地址空间独立”明确区分开。

Linux

路径更长，但系统层次和抽象能力更强。

linux
1. 用户先看到虚拟地址

Linux 程序看到的首先是虚拟地址，而不是硬件上的真实物理地址。

2. MMU / 页表 / TLB 负责翻译

MMU 和页表把虚拟地址翻译成物理地址，TLB 负责加速这类映射查找。

3. 隔离与按需映射由此建立

有了页表之后，进程隔离、权限保护、按需映射和缺页处理才真正成立。

4. 还要区分多种地址视角

驱动开发中还要继续区分用户地址、内核地址、物理地址和 DMA 地址。

5. mmap / 共享内存建立在此之上

共享内存、mmap、缺页异常等机制，本质上都建立在虚拟内存模型之上。

6. 内存模型已经完全换代

所以 Linux 世界不是 MCU 裸机多几层，而是整套内存理解方式都变了。

对比观察

裸机和大多数 FreeRTOS 系统直接在物理地址视角下运行，结构简单但隔离能力弱。Linux 通过 MMU 和页表把程序看到的地址与真实物理位置分开，从而实现进程隔离、权限保护、按需映射以及更灵活的内存管理。

小例子说明

例子：访问 UART 寄存器。裸机里通常就是直接访问固定地址；FreeRTOS 中多个任务虽然都可能操作同一片硬件地址，但本质仍在共享物理空间；Linux 中用户程序不能直接碰这些物理寄存器，而是由驱动在内核里映射并控制，用户看到的只是文件接口、系统调用或受控的 mmap 视图。

08 驱动架构与软件分层对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张要把“驱动不只是寄存器读写”讲清楚，重点观察从应用到硬件之间到底隔了几层，以及每一层在三种系统里分别承担什么职责。

为什么学习这个主题

因为很多人刚学驱动时只看到初始化函数和寄存器操作，看不到架构差异。学这一张图，是为了理解裸机为什么应用常和驱动贴得很近，FreeRTOS 为什么常出现任务、驱动和 ISR 的协作结构，以及 Linux 为什么要通过子系统、驱动模型和总线模型把事情拆开。

裸机 Bare-metal

路径短，上层常直接接触底层资源。

bare
1. 应用直接贴近驱动

应用层常直接调用 BSP、HAL、LL 或自写寄存器封装，访问硬件路径很短。

2. 初始化和业务常常混写

同一个模块里既做时钟和外设初始化，也顺手承担部分业务逻辑，这在小项目里很常见。

3. ISR 与驱动紧密绑定

中断服务例程和驱动代码往往紧贴在一起，ISR 里甚至会顺手做一部分业务处理。

4. 分层主要靠工程习惯

是否拆 app、driver、bsp 等层，更多取决于团队习惯，而不是系统强制框架。

5. 模块变多后耦合上升

项目一旦变大，应用、驱动和硬件细节就容易越缠越紧，重构成本也会抬高。

6. 要警惕应用知道太多

真正要问的是：上层代码是不是知道了过多寄存器、时钟和硬件连线细节。

FreeRTOS

事件先交给内核对象，再由任务继续处理。

rtos
1. 应用先被拆成任务

RTOS 引入任务后，应用不再只是直接调驱动，而是先通过任务把职责拆开。

2. 设备服务任务开始出现

常见系统会出现串口任务、采样任务、存储任务等设备服务角色。

3. ISR 只负责快速处理

中断里通常只做快速确认、搬运或发通知，尽量不在 ISR 中堆太多业务。

4. 队列和信号量交接设备事件

设备事件通过队列、信号量和任务通知继续传给服务任务或应用任务。

5. 驱动、服务任务、应用逐步分开

这样一来，底层外设驱动、设备服务层和上层应用任务会形成更清楚的分工。

6. 分层更清楚也更依赖设计

RTOS 不会自动给你完美架构，但它给了你把层次搭清楚的机会。

Linux

路径更长，但系统层次和抽象能力更强。

linux
1. 用户态先通过标准接口发起访问

应用通常先通过文件、设备节点、sysfs 或网络接口发起访问，而不是直接碰寄存器。

2. 系统调用进入内核子系统

请求进入内核后，会先落到 VFS、TTY、NET、Input 等对应子系统。

3. 再进入总线和驱动模型

设备与驱动会在 platform、I2C、SPI、USB 等总线模型下完成匹配与管理。

4. probe / remove / IRQ 挂对位置

驱动实现中的 probe、remove、中断处理和 file operations 等接口，都要挂在正确框架位置。

5. driver core 统一管理设备关系

driver core 负责统一组织 bus、device、driver 之间的关系，提高复用性和一致性。

6. 第一问题先问属于哪个模型

分析 Linux 驱动时，第一件事通常不是看寄存器，而是先判断它属于哪个子系统和总线模型。

对比观察

裸机分层最短，所以访问硬件最快也最直接；FreeRTOS 在应用和硬件之间加入任务调度与同步层，让代码组织更清楚；Linux 则通过子系统、驱动模型和总线模型把用户接口、设备抽象、总线匹配和驱动实现拆开，换来更强的复用性和扩展性。

小例子说明

例子：读取一个 I2C 温度传感器。裸机常由应用直接调用读寄存器函数；FreeRTOS 里采集任务会通过驱动获取数据，再送给显示或通信任务；Linux 中用户程序可能通过设备节点或 sysfs 读取，而内核再沿着 I2C 子系统和具体驱动路径完成真正访问。

09 调试定位与工程维护对比｜STM32 Bare-metal vs FreeRTOS vs Linux

这一张不讲“怎么写”，而讲“出了问题怎么顺着链路找”。重点放在定位手段、分析路径以及工程维护时的排查层次。

为什么学习这个主题

因为学驱动不只是会写初始化代码，还要会找问题。学这一张图，是为了建立一种顺着链路定位的思路：裸机先看波形和寄存器，FreeRTOS 再加上任务和同步关系，Linux 还要同时考虑用户态、内核子系统、驱动回调和调度行为。

裸机 Bare-metal

路径短，上层常直接接触底层资源。

bare
1. 先看串口 / GPIO / 波形

最直接的办法通常是串口打印、GPIO 翻转和示波器波形，它们先告诉你代码走到哪里。

2. 再用 SWD / JTAG 看寄存器

如果问题更底层，就借助 SWD、JTAG 单步跟踪寄存器、栈指针和程序计数器。

3. 出异常就抓 Fault 现场

HardFault、BusFault 或跑飞时，要优先查看异常栈、故障寄存器和出错地址。

4. 回看 map、栈和内存布局

很多偶发问题最后都要回到 map 文件、栈深度、缓冲区重叠和 RAM 使用情况。

5. 顺着硬件链路缩小范围

裸机调试大多沿着时序、寄存器、波形和异常现场一路往下缩小问题范围。

6. 先抓现场再解释原因

真正有效的习惯不是先猜结论，而是先把现场固定住，再倒推问题原因。

FreeRTOS

事件先交给内核对象，再由任务继续处理。

rtos
1. 先判断是不是任务关系问题

在 RTOS 里，很多问题首先要问的是任务有没有被唤醒、有没有抢到 CPU、有没有被卡住。

2. 看任务状态、优先级和栈

调试第一步通常要看任务是 Ready 还是 Blocked，以及优先级设置和栈是否健康。

3. 再查队列 / 信号量 / 事件组

如果系统行为怪异，还要检查同步对象是否堵塞、满了、没释放或形成反转与死锁。

4. 用 runtime stats / trace 看节奏

运行时统计和 trace 能帮助你看到任务真正的执行比例和系统节奏是否异常。

5. 中断交接链也要一起看

很多问题并不是 ISR 没来，而是通知发了以后任务没有正确接住。

6. 硬件链路和任务链路一起定位

RTOS 调试必须同时看硬件侧和调度侧，不能只盯其中一边。

Linux

路径更长，但系统层次和抽象能力更强。

linux
1. 先看 dmesg / printk / dev_dbg

Linux 定位问题时，最先给线索的常常是内核日志和驱动打印。

2. 再用 ftrace / perf / trace-cmd

如果日志不够，再上 tracing 工具看函数路径、时延热点和关键事件。

3. 查 /proc /sys / debugfs 状态

很多运行时状态、资源分配和设备属性都暴露在 /proc、/sys 和 debugfs 里。

4. 崩溃时再看 kdump / crash

系统崩溃后，还需要借助 kdump、crash 等工具还原崩溃前的上下文。

5. 问题往往横跨多层

Linux 的难点在于问题可能跨越用户态、内核子系统、驱动和调度器，而不是单点失效。

6. 先缩小层次再选工具

最有效的方法通常是先判断问题在哪一层，再选择对应的日志、trace 或崩溃分析工具。

对比观察

裸机定位问题通常是顺着时序、寄存器和波形往下挖；FreeRTOS 除了硬件链路，还要检查任务切换、同步对象和优先级关系；Linux 则必须同时考虑用户态、内核子系统、驱动回调和调度行为，所以工具最多，但分析路径也最长。

小例子说明

例子：串口偶尔丢数据。裸机先看 ISR 是否过长、缓冲区是否溢出、主循环是否来不及消费；FreeRTOS 还要看队列是否满、串口任务优先级是否过低、任务是否长期阻塞；Linux 还得检查驱动缓冲、调度延迟、用户态读取频率以及底半部是否及时运行。

Application Scenes

三种风格的应用场景

这一部分放在 08 章之后，作为整页讲解的落地收束：把八个主题里的结构差异，重新映射到真实项目中的分工、平台能力和选型场景上。

STM32 Bare-metal

适合启动链短、时序强、硬件目标单一的控制型项目，让代码直接贴近外设与寄存器。

典型场景是电机控制板、传感器采集节点、简单工业 IO 控制器、一次只做少量确定性任务的仪器模块。此类项目更在意启动快、路径短、可预测，团队也愿意自己承担状态机、缓存与异常定位的工程复杂度。

STM32 + FreeRTOS

适合“一个 MCU 上并行跑多件事”的中等复杂度系统，把中断、任务和同步对象组织起来。

常见于网关从控板、带显示与通信的控制终端、数据采集记录器、协议转换器、边缘传感控制单元。它比裸机更容易把采样、通信、日志、显示和存储拆成独立任务，但仍保留 MCU 级资源约束与物理地址思维。

Linux / Linux + STM32 协同

适合系统层次多、用户空间丰富、需要网络、文件系统、图形或远程维护能力的复杂平台。

典型场景是工业网关、边缘 AI 设备、人机界面主控、智能终端、Linux 主控加 STM32 实时从控的协同架构。Linux 负责 UI、网络、存储和应用生态，STM32 负责高实时外设控制，两者通过串口、SPI、CAN 或共享内存协议配合完成整机任务。

Selection Notes

选型与协同提示

这一部分继续接在应用场景之后，用来回答“读完 01-08 以后，实际项目中为什么会选某一种模式，或者为什么会采用 Linux + STM32 的协同架构”。

什么时候优先选 Bare-metal

当你最在意启动快、路径短、控制闭环稳定，而且整机功能边界清楚时，Bare-metal 往往是最直接的方案。它适合把 MCU 资源压到极致，但要求开发者自己把中断、缓冲、状态机和异常现场想得非常透。

什么时候切到 FreeRTOS

当系统里已经同时存在采样、通信、显示、日志、存储等多类职责，而且它们需要不同优先级与阻塞唤醒关系时，FreeRTOS 能明显改善代码组织。它不是为了把 MCU 变成 Linux，而是为了把任务协作说清楚。

什么时候上 Linux / 协同架构

当你需要完整网络栈、文件系统、远程维护、图形界面、多进程应用或复杂驱动框架时，Linux 才真正体现优势。如果控制侧还要严格实时，常见做法是 Linux 负责复杂业务，STM32 负责确定性控制，形成主控加从控的协同分工。
